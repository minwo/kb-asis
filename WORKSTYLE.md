# KB AS-IS Workstyle

작업 시작 전에 이 파일을 먼저 읽고 반영한다.

## 목표
- 기존 사용자 코딩 패턴과 최대한 비슷하게 퍼블리싱한다.
- 새로 만든 티가 나는 구조보다 기존 파일들과 닮은 구조를 우선한다.
- Figma/이미지와 다른 부분은 줄이고, 틀릴 수 있는 지점을 먼저 제거한다.

## 요청 해석 원칙
- 사용자가 기준 파일을 말하면 그 파일 구조를 우선 따른다.
- 기준 파일이 없으면 같은 역할의 기존 HTML 1~2개를 먼저 찾고 참고한다.
- 팝업, M 페이지, 상세 페이지, CTA 구조는 반드시 같은 역할의 기존 화면을 먼저 확인한다.
- 확실하지 않은 경우 새 방식 제안보다 기존 방식 재사용을 우선한다.

## 공통 규칙
- 공통 클래스는 `assets/css/comn-form.css` 안에 있는 것만 우선 사용한다.
- `comn-form.css`에 없는 클래스를 임의로 새로 만들지 않는다.
- 인라인 스타일은 추가하지 않는다.
- 단, `table > colgroup > col` 너비 지정용 인라인 스타일은 예외로 허용한다.
- 사용자가 기존에 쓰던 클래스는 임의로 삭제하지 않는다.
- 아이콘은 사용자가 쓰는 기존 태그 패턴을 우선 유지한다.
- `codex-*` 같은 임시성 네이밍은 만들지 않는다.

## 수정 전 확인
- 관련 HTML 1~2개를 먼저 읽는다.
- `assets/css/comn-form.css`에서 사용할 클래스를 먼저 검색한다.
- 사용자가 금지한 항목이 없는지 확인한다.
- 작업이 신규 생성인지, 기존 파일 수정인지 먼저 구분한다.

## 마크업 원칙
- 제목, 본문, 설명, CTA 순서와 간격은 기존 화면 결을 우선한다.
- 버튼은 같은 역할의 기존 버튼 구조를 복제에 가깝게 맞춘다.
- 표 대신 정보 박스가 더 기존 패턴에 맞으면 박스를 쓴다.
- `data-row`보다 사용자가 따로 지정한 구조가 있으면 그 구조로 통일한다.
- 문장 줄바꿈이 중요하면 `p` 여러 개보다 `br` 사용 여부를 먼저 확인한다.

## 팝업 원칙
- 팝업은 먼저 기존 팝업 파일을 기준으로 잡는다.
- 헤더, 닫기 버튼, 본문, CTA 구조를 기준 파일과 동일하게 맞춘다.
- 팝업 컨텐츠만 바뀌는 경우 뼈대는 최대한 그대로 둔다.

## 신규 파일 생성 원칙
- 새 HTML 생성 시 `coding-list.meta.json`에 함께 등록한다.
- 등록 후 `scripts/sync-coding-list.ps1`를 실행해 `coding-list.html`까지 반영한다.
- 코딩리스트 정렬은 내림차순 기준을 유지한다.

## 수정 시 주의
- 사용자가 직접 만든 설정이나 코드 스타일은 임의로 되돌리지 않는다.
- `assets/css/comn-form.css` 수정은 꼭 필요한 경우에만 하고, 먼저 사용자 의도를 확인한다.
- 클래스가 안 먹으면 새 클래스 추가보다 기존 공통 클래스(`gap-x-xs`, `gap-x-s` 등)로 대체 가능한지 먼저 본다.
- Figma와 다른 부분이 보여도 바로 새 클래스 추가로 해결하지 않는다.

## Self-check
- 기준 파일을 먼저 확인했는가
- 새 클래스를 임의로 만들지 않았는가
- 인라인 스타일이 남아 있지 않은가
- `comn-form.css` 안의 클래스만 우선 사용했는가
- 아이콘 태그 패턴이 기존과 같은가
- 간격, 색상, 타이포를 한 번 더 비교했는가
- 새 HTML이면 coding-list까지 반영했는가

## 요청 템플릿
아래 형식으로 요청이 오면 정확도를 더 높게 맞춘다.

```text
작업 종류: 신규 / 수정
기준 파일: 예) DH03_LT_BF0134P_DA.html
대상 파일: 예) DH03_LT_BF0139P_DA.html
스타일 규칙: comn-form만 사용, 새 클래스 금지
허용 범위: 텍스트 / 마크업 / JS
금지 범위: 인라인 스타일, 공통 CSS 수정
비교 기준: 첨부 이미지 / Figma URL
```

## 응답 방식
- 작업 전 무엇을 확인하고 무엇을 수정할지 짧게 설명하고 시작한다.
- 수정 후에는 무엇을 바꿨는지 파일 기준으로 간단히 정리한다.
